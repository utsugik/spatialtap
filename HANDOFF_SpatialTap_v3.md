- **核の操作は「タップ＝音がそこへ行く」の1動作。** 高度な機能はこれを邪魔しない範囲で。

## 3. 実装済みの機能（index.html）

- 動画読み込み→画面タップ/ドラッグ地点に音源定位（Web Audio PannerNode + HRTF）。タップ地点は保持、次タップで切替。
- 画面 横=方位(±90°) / 縦=高さ(±45°)、距離は Distance フェーダー。
- モノ/ステレオ自動判定（decode成功時 buffer.numberOfChannels）。モノ=1点、ステレオ=L/R 2点。
- Width（L/R間隔）、Room（合成IRリバーブの長さ・量）、yaw/pitch/roll（L/Rペアを3D回転・無限回転・ダブルタップで0）。
- 右上「LOG」ボタン: iPhoneでも状態/エラーを画面表示（デバッグ用）。

---

## 4. 音響設計の要点（なぜこの実装か）

- ステレオは ChannelSplitter で L/R 分離→別々の PannerNode。左右が別の点として定位し幅・回転を正しく扱える。
- 3D座標は center() + rotOffset() の1か所で計算。視覚・音響・数値が必ず一致。
- Web Audio 座標系: 前方=-z、右=+x、上=+y。listener は forward(0,0,-1)/up(0,1,0)。
- iOS対策: 動画はミュート再生（映像専用）、音声は decodeAudioData で空間化。createMediaElementSource の iOS 無音バグを回避。デコード失敗時のみ element 音声にフォールバック。

## 5. 判明済みの問題と原因

### (A) Claudeアプリ内プレビュー（WebView）では動画が再生できない ← 最大の障壁
- H.264 の .mov でも `VIDEO ERROR code=4` が出る。原因はコーデックでなくWebViewの再生制限。
- 対策: 実ブラウザ（iPhone Chrome/Safari 本体）で開く=HTTPS URL必須=ホスティング必須。
- ※未検証: 実ブラウザ本体での .mov(H.264/HEVC) 再生可否は GitHub Pages 公開後に確定。

### (B) コーデック
- 画面収録/高効率設定は HEVC/.mov。「設定→カメラ→フォーマット→互換性優先」でH.264（切り分け用。最終形では強制しない）。
- iOS ChromeもWebKitのため「Chrome/Safari本体」の検証は実質1エンジンで済む。HEVCはApple自身の形式なので実ブラウザでは再生できる見込みが高い（※要検証・仮説）。

### (C) ノブが動かない（解決済み）
- Pointer Events + setPointerCapture、子要素 pointer-events:none、pointerdown で stopPropagation。実機確認済み。

### (D) 動画が「no video」のまま（原因確定）
- 選択は伝わっている（LOG確認済み）。表示されないのは (A) のWebView制約が主因。

## 6. 次にやること（優先順・12章の決定を反映）

1. ✅ **完了（2026-07-10）: GitHub Pages 公開 → iPhone 実ブラウザで動作確認。**
   - 公開URL: https://utsugik.github.io/spatialtap/ （Publicリポジトリ: https://github.com/utsugik/spatialtap ）
   - 実機検証結果: `VIDEO ERROR` なし、`audio decoded` ch=2 dur=189.4 sec（.mp4/H.264）で成功。
   - **確定した結論: 検証したファイル（.mp4/H.264）は実ブラウザで問題なく再生・decode可能。ffmpeg.wasmは現時点で不要（12章「非採用」のまま据え置き）。**
   - ※注記: 今回検証したのは .mp4/H.264。HEVC/.mov単体での再生可否はまだ未検証のまま（5章(B)参照、今後実際にHEVC素材を使う場面が来たら確認）。
2. ✅ **完了（2026-07-10）: 音の説得力3点セット**（index.html 実装済み・未pushの実機再検証はこれから）。
   - (a) 距離連動の dry/wet 比: panner→convの直結を独立GainNode `send` 経由に変更。距離0.4〜12mを send.gain 0.15〜1.0にマッピングし、遠いほどwetが増える構成にした。
   - (b) 出力リミッター: dry/wetの合流前に `DynamicsCompressor`（threshold -3dB, ratio 20, attack 3ms, release 150ms）を追加。
   - (c) Room フェーダーのデバウンス: `fRoom` の input イベントで `makeIR()` を直接呼ばず、150ms の `setTimeout` 経由に変更（ドラッグ中の多重生成を防止）。
   - ※距離連動ローパス・IR初期反射（12章「採用」の別項目）は今回のスコープ外。次の作業候補として残っている。
   - **不具合と修正（2026-07-10）: 実機検証で、距離に関わらず常時「パリパリ」というクラックリングノイズが発生。原因はオーディオグラフの二重接続ではなく（確認済み・問題なし）、リミッターがヘッドルーム不足で追いつかずAudioDestinationNode手前で瞬間クリップしていたためと判断（電子ドラム等の速い立ち上がりで顕著）。対策として dry ゲインを1→0.6（-4dB相当）に下げてヘッドルームを確保し、リミッターも threshold -3→-9dB・knee 0→6・ratio 20→8・attack 3ms→5ms・release 150ms→200msに変更し、急な潰し方をやめて早め・穏やかに効かせる設定にした。**
   - **次にやること: この修正版を実機（iPhone、電子ドラム含む素材）で再生し、クラックリングが解消したか、distance/roomの挙動に副作用がないかを確認する（ENV）。**
3. **モノ音声フォールバック時の R チャンネル無音バグ修正**（小さいが「壊れてる」誤解を招くため優先）。
4. （保留・体験拡張）デモ動画同梱＋Orbit（自動旋回）ボタン、ジャイロ連動、軌跡の記録/再生。
5. （保留・要実機調整）高さ方向のEQ強化、方位マッピングの実測合わせ（±35°/±90°の切替）。

※ ffmpeg.wasm 変換は 1 の結果次第で不要になる可能性が高い。着手は保留。

## 8. 既知の限界・確認が必要な点（正直な注記）

- 高さ(上下)の聴感は汎用HRTFのため弱め（個人差）。強化は PHASE/実測HRTF が必要。
- 平面動画のため真後ろは非対応（前方半球）。360°は別アプローチ（球体版/Three.js）。
- decode方式は映像と音声が別系統→長尺ループで微小ズレの可能性（短尺は問題なし）。
- リバーブは合成IR（減衰ノイズ）で初期反射なし＝「広さ」でなく「残響長」だけが変わる状態（12章で改善対象）。
- **普通の動画の音声は環境音・声などが混ざった1本のミックス。「動画内の特定の物だけを分離」はできず「音声全体を配置」する体験。** デモ/説明で期待値を正しく設定する必要あり。
- スピーカー再生では効果がほぼ消える → **UI最初にヘッドホン必須を明示**（離脱の最大リスク回避）。
- 数値提案（例: 8kHz±3〜4dB、FOV±30〜35°）は方向性は妥当だが未検証の経験則。実機で聴きながら調整前提。
- HEVC実機再生可否・decodeAudioData可否・ffmpeg.wasm要否・その実機性能はいずれも未検証。

## 12. 決定事項（2026-07-10 の監査・査読の結論）

Fable 5 が監査 → Sonnet 5 が査読（妥当な指摘と未検証の経験則を仕分け）→ 現行モデルがレビュー。結論を確定として記録する。

### 採用（実装する。優先度は 6章のとおり）
- **距離の dry/wet 比バグ修正**（最重要）。現状は距離を変えても直接音/残響比が一定＝実質音量フェーダー。convへの送りを独立 GainNode にし、距離に応じて wet を増やす。※行数は少なくても「信号経路の設計変更」である点に注意。
- **出力リミッター**（DynamicsCompressor）。distance最小時のゲイン増で歪む危険への対策。第一印象の「壊れてる」を防ぐ。
- **Room フェーダーの makeIR デバウンス**（約150ms）。ドラッグ中の多重生成でプチノイズ/処理落ちを防ぐ。
- **モノ音声フォールバック時の R 無音バグ修正**。フォールバック時はモノ扱いをデフォルトに。
- **距離連動ローパス**（遠いほど高域減衰）＋**IR に初期反射を加える/後半の高域を削る**（距離・空間サイズの説得力向上）。
- **UIにヘッドホン必須を最初に明示**。離脱防止。
- **メモリ対策**: ファイルサイズ上限チェック＆警告、URL.createObjectURL の revoke。iOSのタブクラッシュ防止。
- **初回再生の音量配慮**（控えめ〜フェードイン）※現行モデル追加提案。イヤホン直の不快な第一印象を防ぐ。

### 保留（土台が実機で動いてから／体験拡張）
- デモ動画同梱＋Orbit（自動旋回）ボタン。※ページを重くしない軽量デモにする。
- ジャイロ連動（DeviceOrientationEvent.requestPermission。許可前に理由を一言添える）。
- 軌跡の記録・再生。空間化動画の書き出し（MediaRecorder、iOS実用性は要検証）。
- 高さ方向のEQ強化（8kHz付近ブースト等）と方位マッピングの±35°/±90°切替。※数値は実機で聴いて調整。

### 非採用/現時点で着手しない
- ffmpeg.wasm によるブラウザ内変換（30MB級DLで「気軽さ」と矛盾）。※6章-1の実機検証で要否が確定するまで着手しない。
- アンビソニックス（Omnitone等）への置換。回転は数学的に自然だが実装複雑度大。将来の360°版で検討。
- index.html と v1_room の重複一本化。※統合(4-保留)を実施するときのみ。今は触らない。
